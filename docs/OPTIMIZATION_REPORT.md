## 二、优化建议（按优先级）

### P0 - 关键（建议立即修复）

| 问题 | 位置 | 建议 |
|------|------|------|
| 每日亏损追踪不完整 | `risk_manager.py` | `update_loss()` 仅在平仓时调用，需确保每次平仓都正确更新 |
| 风险状态不持久 | `risk_manager.py` | 重启后 daily_loss、consecutive_losses 重置，可考虑写入文件 |
| Web 控制面板无认证 | `web/app.py` | 添加 Basic Auth 或 API Key 鉴权 |
| 配置未校验 | `config_loader.py` | 加载时校验杠杆、百分比等范围 |
| 订单成交未验证 | `trade_executor.py` | 设置 TP/SL 前确认开仓订单已成交 |

### P1 - 高优先级

| 问题 | 建议 |
|------|------|
| 错误处理过于宽泛 | 使用自定义异常，区分可重试/不可重试 |
| 缺少输入校验 | 校验 quantity、symbol、price 合法性 |
| 加仓时仓位计算 | 考虑已有持仓，避免超限 |
| 数据新鲜度 | 检查 K 线时间戳，拒绝过期数据 |
| 重试策略 | 指数退避替代固定 5 秒延迟 |

### P2 - 中优先级

| 问题 | 建议 |
|------|------|
| 串行获取市场数据 | 使用 `concurrent.futures` 并行请求 |
| 无市场数据缓存 | 添加 TTL 缓存，减少 API 调用 |
| 缺少 K 线数据 | Prompt 中补充最近 18 根 K 线 OHLC |
| 指标数据不足 | 校验 DataFrame 行数再计算 RSI 等 |
| 健康检查 | 添加 `/health` 等自检接口 |

### P3 - 低优先级

| 问题 | 建议 |
|------|------|
| 单一数据源 | 考虑多交易所或备用数据源 |
| 无回测框架 | 引入 backtrader 或自建回测 |
| 无性能指标 | 计算 Sharpe、最大回撤等 |
| 通知能力 | 集成 Telegram/邮件告警 |
| 部分平仓 | 支持按比例减仓 |

## 三、架构与代码质量

**优点**：模块划分清晰，配置与逻辑分离，Web 面板实用。

**待改进**：
- 引入结构化日志（logging 模块 + 级别）
- 为关键函数添加类型注解
- 增加单元测试（至少覆盖 trade_executor、decision_parser）
- 敏感信息脱敏（日志中不输出 API 响应体）

---

## 四、安全建议

1. **Web 面板**：增加认证，限制访问 IP 或使用反向代理鉴权
2. **API Key**：避免在日志中打印，考虑使用环境变量或密钥管理服务
3. **Symbol 校验**：Web 关单接口校验 symbol 白名单
4. **请求限流**：对 Web API 做简单限流，防止滥用
